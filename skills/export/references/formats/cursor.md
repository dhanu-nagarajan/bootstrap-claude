# Cursor Export Format

## Output Structure

Generate multiple `.mdc` files in the `.cursor/rules/` directory. Create the directory if it does not exist.

## File Format

Each `.mdc` file uses YAML frontmatter followed by markdown content:

```
---
description: [concise description of when this rule applies]
globs: ["*.ts", "*.tsx"]
alwaysApply: false
---

# Rule Title

[Rule content in markdown]
```

### Frontmatter Fields

- **description** — One sentence describing when Cursor should activate this rule. Written as a condition: "When writing TypeScript files" or "When modifying test files."
- **globs** — Array of file glob patterns. Omit entirely for `alwaysApply: true` rules.
- **alwaysApply** — Set to `true` only for rules that apply universally (architecture, security, forbidden patterns). Default `false`.

## File Split Strategy

Generate these files, adapting to what the project actually uses:

| File                | alwaysApply | globs                           | Source                          |
| ------------------- | ----------- | ------------------------------- | ------------------------------- |
| `architecture.mdc`  | true        | —                               | architecture.md, core.md        |
| `{language}.mdc`    | false       | Language file extensions         | language.md                     |
| `testing.mdc`       | false       | `["*.test.*", "*.spec.*", ...]` | testing.md                      |
| `security.mdc`      | true        | —                               | security.md                     |
| `forbidden.mdc`     | true        | —                               | forbidden.md                    |
| `error-handling.mdc`| false       | Language file extensions         | error-handling.md               |
| `components.mdc`    | false       | `["*.tsx", "*.jsx", "*.vue"]`   | components.md (if UI project)   |

Only generate files for which source material exists. Skip `components.mdc` for non-UI projects, etc.

### Naming the Language File

Use the project's primary language for the filename: `typescript.mdc`, `python.mdc`, `rust.mdc`, `go.mdc`, etc.

## Content Guidelines

- **Max ~200 lines per file.** Cursor loads matched rules into context on every request — bloated rules waste the context window.
- **Use bullet lists over prose.** Cursor rules are scanned, not read end-to-end.
- **Be directive.** "Use X" not "Consider using X" or "It is recommended to use X."
- **Include code patterns** only when the rule is ambiguous without one. Keep examples to 3-5 lines max.
- **No cross-references** between rule files. Each file must be self-contained because Cursor loads them independently.

## Forbidden Patterns File

The `forbidden.mdc` file is critical. Structure it as a flat list:

```markdown
# Forbidden Patterns

These patterns must never appear in this codebase.

- NEVER use `any` type — use `unknown` and narrow
- NEVER use default exports — use named exports
- NEVER catch errors silently — always log or rethrow
- NEVER commit `.env` files — use `.env.example`
[...]
```

Keep each rule to one line. No explanations — if a developer needs to know why, they can check the `.claude/` system.

## Handling Existing Cursor Rules

If `.cursor/rules/` already exists with user-created rules:
- Do NOT overwrite files that were not generated by this export
- Prefix generated files with a comment: `<!-- Generated by bootstrap-claude /export -->`
- If a generated filename conflicts with an existing file, append `-exported` to the name
