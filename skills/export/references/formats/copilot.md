# GitHub Copilot Export Format

## Output Structure

Single file: `.github/copilot-instructions.md`

Create the `.github/` directory if it does not exist.

## Context Budget

Copilot has a smaller context budget than Claude Code and does not perform agentic workflows. Focus on rules that help with **code completion and inline suggestions**:

- Naming conventions and code style
- Type usage and signatures
- Common patterns and idioms
- What NOT to generate (forbidden patterns)

Omit workflow rules, pre-commit checklists, and process guidelines — Copilot does not execute multi-step tasks.

## File Structure

```markdown
# Project Coding Standards

## Architecture
[2-5 bullet points on module structure and dependency rules]

## Naming Conventions
[How to name files, functions, variables, types, constants]

## Code Style
[Language-specific conventions: imports, error handling, typing]

## Patterns
[Common patterns used in this project — e.g., repository pattern, error boundaries]

## Forbidden Patterns
[Flat bullet list of things to NEVER do]

## Testing
[How to write tests: framework, file naming, assertion style]
```

## Content Guidelines

- **Target 150-300 lines total.** This file is included in every Copilot interaction.
- **Bullet lists only.** No prose paragraphs, no long explanations.
- **One rule per line.** Each bullet should be a complete, actionable directive.
- **Skip methodology.** No mention of "protocols", "personas", "checklists" — Copilot does not understand these concepts.
- **Concrete over abstract.** "Use `Result<T, AppError>`" is better than "Use typed error handling."
- **Include naming examples** inline: "Functions: `camelCase` (`getUserById`, `validateInput`)."

## Forbidden Patterns Section

Always include this section, even if it means the file runs slightly long. Structure as a flat list with the NEVER keyword:

```markdown
## Forbidden Patterns

- NEVER use `any` type
- NEVER use default exports
- NEVER disable linting rules inline
- NEVER use `console.log` in production code — use the logger
```

## Handling Existing File

If `.github/copilot-instructions.md` already exists:
- Read it first to check for user-written content
- Add a header comment: `<!-- Generated by bootstrap-claude /export -->`
- If the existing file has custom sections not covered by the export, preserve them at the bottom under a `## Project-Specific` heading
