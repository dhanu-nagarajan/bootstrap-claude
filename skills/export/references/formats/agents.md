# AGENTS.md Export Format

## Output Structure

Single file: `AGENTS.md` at project root. Optionally, additional `AGENTS.md` files in key subdirectories if the project has distinct subsystems with different conventions.

## Context Budget

AGENTS.md is a convention used by multiple AI tools (GitHub Copilot Workspace, Devin, and others). It can be larger than single-tool formats because it is typically loaded selectively by directory.

## File Structure

```markdown
# AGENTS.md

## Project Overview
[What this project does, primary language and framework, key dependencies]

## Architecture
[Directory structure with descriptions]
[Module boundaries and dependency rules]
[Data flow overview]

## Development Standards

### Naming Conventions
[Files, functions, variables, types, constants]

### Code Style
[Language-specific patterns, import conventions, formatting]

### Error Handling
[Error types, propagation, logging]

### Security
[Auth patterns, input validation, secrets handling]

## Testing
[Framework, conventions, how to run, coverage expectations]

## Forbidden Patterns
[Anti-patterns that must never appear]

## Build & Run
[All commands: build, test, lint, dev server, deploy]

## Contributing
[How to add features, where to put new code, PR conventions]
```

## Content Guidelines

- **Target 200-500 lines** for the root file. Subdirectory files should be 50-150 lines.
- **Include structural information.** Unlike Aider (which has repo-map), agents reading AGENTS.md need explicit directory descriptions.
- **Include commands.** Build, test, lint, and deploy commands.
- **Include contributing guidance.** AGENTS.md readers are often performing substantial tasks — they need to know the project's workflow.
- **Be tool-agnostic.** Do not reference Claude, Copilot, or any specific tool. Write for a generic AI coding agent.

## Subdirectory AGENTS.md Files

If the project has distinct subsystems (e.g., `frontend/`, `backend/`, `infra/`), consider generating per-directory files with rules specific to that subsystem. Each subdirectory file should:

- Focus only on that directory's conventions
- Not repeat root-level rules
- Reference the root AGENTS.md for general standards

Only generate subdirectory files if the project has clearly distinct subsystems with different conventions. Do not create them for simple projects.

## Handling Existing File

If `AGENTS.md` already exists:
- Read it first
- Check if it was previously generated (`<!-- Generated by bootstrap-claude /export -->`)
- If previously generated, overwrite
- If user-written, preserve and append exported content under a clearly marked section
